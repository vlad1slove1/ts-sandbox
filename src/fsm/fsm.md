# Конечные автоматы (Finite State Machines)

## Определение

**Конечный автомат** (Finite State Machine, FSM) — математическая модель вычислений, описывающая систему, которая в любой момент времени находится ровно в одном из конечного числа состояний и переходит между ними в зависимости от входных символов (событий).

## Основные понятия

### Компоненты FSM

- **Состояния (States)** — конечное множество возможных положений системы
- **Начальное состояние** — состояние, в котором автомат начинает работу
- **Конечные (принимающие) состояния** — состояния, в которых автомат может завершить работу
- **Алфавит (входные символы)** — множество допустимых входных воздействий
- **Функция переходов** — правило, определяющее следующее состояние по текущему состоянию и входному символу

### Типы конечных автоматов

| Тип | Описание |
|-----|----------|
| **Детерминированный (DFA)** | Для каждой пары (состояние, символ) существует ровно один переход |
| **Недетерминированный (NFA)** | Возможны несколько переходов или отсутствие перехода для пары (состояние, символ) |

## Математическая модель

Конечный автомат задаётся кортежем из пяти элементов:

```
M = (Q, Σ, δ, q₀, F)
```

Где:
- **Q** — конечное множество состояний
- **Σ** — конечный входной алфавит
- **δ** — функция переходов: Q × Σ → Q (для DFA)
- **q₀** — начальное состояние (q₀ ∈ Q)
- **F** — множество конечных состояний (F ⊆ Q)

## Диаграмма состояний

```
                    ┌─────────────┐
                    │   Начало    │
                    └──────┬──────┘
                           │
                           ▼
              ┌────────────────────────┐
     ┌────────│   Состояние A          │────────┐
     │        └────────────────────────┘        │
     │ event1                          event2   │
     │        ┌────────────────────────┐        │
     └───────►│   Состояние B          │◄───────┘
              └────────────────────────┘
                           │
                      event3
                           │
                           ▼
              ┌────────────────────────┐
              │   Конечное состояние   │
              └────────────────────────┘
```

## Пример: автомат для двери

```
Состояния: {Закрыта, Открыта}
События: {Открыть, Закрыть}

     Открыть          Закрыть
Закрыта ──────► Открыта ◄────── Закрыта
   ▲                │
   │    Закрыть     │  Открыть
   └────────────────┘
```

## Применение в программировании

### 1. Парсинг и валидация

- Проверка форматов (email, телефон, URL)
- Лексический анализ (токенизация)
- Разбор протоколов

### 2. Управление состоянием

- Жизненный цикл сущностей (заказ: создан → оплачен → отправлен → доставлен)
- Состояния UI-компонентов (idle → loading → success/error)
- Управление подключениями (disconnected → connecting → connected)

### 3. Игровая логика

- AI-поведение персонажей
- Управление сценами и уровнями
- Обработка ввода

### 4. Сетевые протоколы

- TCP: установка и разрыв соединения
- HTTP: жизненный цикл запроса
- WebSocket: управление соединением

## Реализация в TypeScript

```typescript
type State = 'idle' | 'loading' | 'success' | 'error';
type Event = 'fetch' | 'resolve' | 'reject';

const transitions: Record<State, Partial<Record<Event, State>>> = {
  idle: { fetch: 'loading' },
  loading: { resolve: 'success', reject: 'error' },
  success: { fetch: 'loading' },
  error: { fetch: 'loading' },
};

class StateMachine {
  private state: State = 'idle';

  transition(event: Event): void {
    const nextState = transitions[this.state][event];
    if (nextState) {
      this.state = nextState;
    }
  }

  getState(): State {
    return this.state;
  }
}
```

## Преимущества

- **Явная модель** — все состояния и переходы видны и документированы
- **Предсказуемость** — поведение системы полностью определяется текущим состоянием и событием
- **Тестируемость** — легко покрыть все комбинации (состояние × событие)
- **Отладка** — просто отследить, в каком состоянии находится система
- **Визуализация** — диаграммы состояний наглядно показывают логику

## Недостатки и ограничения

- **Сложность при росте** — число состояний и переходов может расти экспоненциально
- **Только дискретные системы** — не подходит для непрерывных процессов
- **Один активный контекст** — классический FSM не хранит историю (решается иерархическими автоматами)

## Связь с паттернами проектирования

- **State** — объектно-ориентированная реализация конечного автомата; каждое состояние — отдельный класс
- **Strategy** — похожая структура, но Strategy выбирается извне, а State меняется по событиям
- **Command** — события/действия, вызывающие переходы в FSM

## Расширения

- **Иерархические автоматы (HFSM)** — состояния могут содержать под-автоматы
- **Автоматы с выходом (Mealy/Moore)** — генерируют выходные символы при переходах
- **UML Statecharts** — расширенная нотация с параллельными регионами, история и т.д.
